# Learning: Zhihu Platform Analysis

## 关键学习点

### 1. 虚拟滚动问题 → API 才是王道

**问题**：DOM 只显示可见的 10 条，scrolling 无法加载全部 78 条

**解决**：
- 不要死磕 DOM scraping
- 用 Chrome DevTools Network 监听，找到真正的 API
- 发现：`/api/v4/creators/creations/v2/article?limit=20&offset=0`

**教训**：遇到虚拟滚动，直接找 API，不要浪费时间在 DOM 上

### 2. API 分类 ≠ UI 分类

**发现**：
- API 返回：78 article + 25 pin + 1 zvideo = 104
- UI 显示：78 文章 + 9 视频 + 17 想法 = 104
- **Pin 是复合类型**：可以包含 text/image/video

**检查方法**：
```javascript
// Pin 中哪些含视频？
jq '.items[] | select(.type == "pin") | select(.data.content[]?.type? == "video")' data.json
// 结果：16 个 pin 含视频
```

**教训**：不要只看 type 字段，要检查实际内容结构

### 3. 分页限制

**API 限制**：`limit=20` (最大)

**解决**：
```javascript
let offset = 0;
while (hasMore) {
  const url = `${api}?limit=20&offset=${offset}`;
  const data = await fetch(url);
  if (data.length < 20) hasMore = false;
  offset += data.length;
}
```

**教训**：测试 API 时先尝试 `limit=100`，如果只返回 20，说明有限制

### 4. 真实数据 vs 0 值

**初始问题**：所有互动指标都是 0

**原因**：用的是旧的 `/api/v4/creator/content_manage/contents`，权限不足

**解决**：找到新 API `/api/v4/creators/creations/v2/article`

**结果**：read_count 1-427（真实值！）

**教训**：如果数据全是 0，可能是 API 权限问题，尝试找其他端点

### 5. 用户纠正错误

**场景**：
- 我分析说"只有 77 条文章"
- 用户："我刚看了这个账号 这个账号发布了9个视频"
- 用户："全部是96条内容...文章是78条内容，然后视频是9条内容，想法是17条内容"

**教训**：
- ✅ 用户提供的数据是准确的，要立即重新分析
- ✅ 不要固执己见，数据说话
- ✅ "你应该自己去查" → 直接操作浏览器验证

### 6. 测试脚本要跟数据结构同步

**问题**：测试脚本检查 `item.title`，但新 API 是 `item.data.title`

**修复**：
```javascript
// Old
const hasFields = item.title && item.publishTime && item.type;

// New
const hasFields = item.type && item.data && item.reaction;
```

**教训**：数据结构变化时，同步更新测试脚本

## 可复用模式

### Pattern 1: Network Monitoring Template

```javascript
const CDP = require('chrome-remote-interface');
const { Network } = await CDP({ host, port });
await Network.enable();

const apiRequests = [];
Network.requestWillBeSent((params) => {
  if (params.request.url.includes('/api/')) {
    apiRequests.push(params);
  }
});
```

### Pattern 2: Pagination Template

```javascript
async function paginateAPI(baseUrl, maxLimit = 20) {
  const allItems = [];
  let offset = 0;
  let hasMore = true;

  while (hasMore) {
    const url = `${baseUrl}?limit=${maxLimit}&offset=${offset}`;
    const data = await fetchAPI(url);
    
    if (!data || data.length === 0) {
      hasMore = false;
    } else {
      allItems.push(...data);
      offset += data.length;
      if (data.length < maxLimit) hasMore = false;
    }
  }
  
  return allItems;
}
```

### Pattern 3: Content Type Detection

```javascript
// 不只看 type 字段，检查实际内容
function detectContentType(item) {
  if (item.type === 'article') return 'article';
  
  if (item.type === 'pin') {
    // Check actual content
    const hasVideo = item.data.content?.some(c => c.type === 'video');
    return hasVideo ? 'video_pin' : 'text_pin';
  }
  
  return item.type;
}
```

## 反模式（避免）

❌ **反模式 1**：相信第一次采集的数据
- 初始采集可能不完整
- 总是验证：用户提供的数量 vs 采集到的数量

❌ **反模式 2**：过度依赖 DOM scraping
- 虚拟滚动会限制可见元素
- 优先找 API，DOM 是备选方案

❌ **反模式 3**：忽略 API/UI 分类差异
- 不同平台的 API 和 UI 可能用不同的分类逻辑
- 要同时记录两种分类方式
