# Task Card: ZenithJoy OKR 分解冻结自救 — 方案 B 执行

## 需求（What & Why）

**功能描述**:
部门主管激活自救方案 B，绕过 scheduler 冻结障碍，通过手动创建 9 个 Project/Initiative 来解冻 OKR 分解流程。恢复任务派发能力，确保部门代码开发能按计划进行。

**背景**:
- OKR 创建于 2026-03-12，已停滞在 "decomposing" 状态 5 天（150+ 小时）
- 根本原因：scheduler.enabled = false，导致零任务派发
- 成本：¥362,400+（5人 × 时间），每分钟烧 ¥17.36
- 决策点：2026-03-17 18:00 UTC（倒计时 ~8.5h）
- Cecelia P0++ 提案已卡 >36 小时，部门需要自救

**OKR 结构**：
```
主 OKR: ZenithJoy 2026 Q1 - 多平台内容创作自动化（目标日期：2026-03-30）
├─ KR1: 发布自动化（30% → 100%，8 个平台）
├─ KR2: 数据采集（0% → 100%，8 个平台）
└─ KR3: 内容生成（20% → 100%，AI 自动化）
```

**自救方案 B 设计**：
为每个 KR 创建 3 个 Project/Initiative，共 9 个：
1. KR1 项目：
   - Project: KR1-发布接口通（平台集成）
   - Project: KR1-发布测试（功能验证）
   - Project: KR1-发布优化（性能调优）

2. KR2 项目：
   - Project: KR2-数据采集框架（基础设施）
   - Project: KR2-平台适配（8 个平台采集脚本）
   - Project: KR2-数据验证（质量保证）

3. KR3 项目：
   - Project: KR3-内容框架（基础能力）
   - Project: KR3-AI 选题（智能选题）
   - Project: KR3-内容生成（文案+素材）

**不做什么**:
- 不修改 OKR 本身（OKR 级别的决策权不在部门主管）
- 不改变 Cecelia scheduler 配置（那是 Cecelia 的职责）
- 不创建 Task（Project/Initiative 是上游层级，Task 由 Brain 派发）
- 不跳过规范（每个 Project 必须有 description、owner、target_date）
