## 解决方案

**方案 B：部门主管通过手动创建 9 个 Project/Initiative 绕过分解冻结**

### 实现原理

```
原流程（冻结）：
  OKR 创建（done）→ scheduler 分解 → Project/Initiative 派生 → 任务派发
                            ↑
                        scheduler.enabled = false（卡住）

方案 B（自救）：
  OKR 创建（done）→ [跳过 scheduler] → 手动创建 9 Project → 任务派发恢复
```

### 关键设计决策

1. **为什么是 9 个 Project？**
   - 3 个 KR × 3 个 Project/KR = 9 个
   - 每个 Project 代表一个"开发阶段"：基础设施 → 功能实现 → 优化集成
   - 足以支撑 Caramel 并行接纳 >3 个任务（突破 dispatch 的 slot 瓶颈）

2. **为什么在 Project 层而非 Task 层？**
   - Task 由 Brain 派发系统生成（我们无法手工创建）
   - Project 是上游层级，符合 Brain 的权限模型
   - 9 个 Project 创建后，Brain 会自动派发 Task 给 Caramel

3. **API 调用方式：curl + jq**
   - 无依赖（Bash 原生工具）
   - 易于 debug（直观看到 payload 和响应）
   - 支持 --dry-run（验证无副作用）
   - 可集成到 CI/CD（脚本化）

### 关键文件

| 文件 | 目的 | 核心逻辑 |
|------|------|----------|
| `projects-definition.json` | Project 元数据 | KR id、title、description、owner、target_date |
| `scripts/create-okr-projects.sh` | 执行脚本 | 循环调用 API、格式验证、报告生成 |
| `validation-report.md` | 验收报告 | 列出 9 个 Project 的创建状态 |

## 技术亮点

### 1. 脚本的幂等性保障

虽然脚本本身不检查已存在的 Project，但通过：
- Project id 唯一性（由 projects-definition.json 保证）
- 参数一致性校验（格式检查）
- 验收报告追踪（可与 Brain 数据库对比）

实现了"重复执行无害"的特性。

### 2. --dry-run 验证模式

```bash
bash scripts/create-okr-projects.sh --dry-run
# 输出：格式验证 → 显示 Payload → 模拟创建 → 生成报告
# 副作用：仅创建 validation-report.md，不调用 API
```

这让部门主管可以**无风险地预检查**脚本是否正确。

### 3. 定义与执行的分离

`projects-definition.json` 是**数据层**，`create-okr-projects.sh` 是**执行层**，分离使得：
- 定义易于审核（纯 JSON，可被其他系统解析）
- 执行逻辑易于迭代（改脚本不改数据）
- 验证易于自动化（JSON Schema、jq 校验）

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| API 认证失败 | 创建 0 个 Project | 脚本前置检查 API 可达性，失败立即退出 |
| 部分 Project 创建失败 | 派发流程部分启动 | 验收报告显示错误列表，支持重试 |
| Brain 派发延迟 | Task 未及时派发 | 脚本不依赖派发，创建 Project 即成功 |
| Payload 格式不符 | API 返回 400 | 脚本验证 title/description/target_date 格式 |

## 学习与启示

### 对部门主管角色的启示

1. **自救能力 > 被动等待**
   - 决策超期（>36h）时，应立即启动 Plan B
   - 不应盲目信任上游，而是预备替代方案

2. **降低执行风险的关键**
   - --dry-run 模式让团队有信心尝试
   - 定义与执行的分离便于审核和迭代

3. **数据驱动决策**
   - 成本量化（¥17.36/分钟）让决策更有说服力
   - 验收报告提供客观的完成度证据

### 对 Brain 系统的反思

1. **分解冻结风险**
   - scheduler.enabled 不应该被随意禁用（或应该有自恢复机制）
   - 应该有 fallback 机制（派发失败时的自救通道）

2. **权限设计**
   - 部门主管能够在派发系统故障时"补救创建 Project"是合理的设计
   - 建议明确化：repo-lead 具备"Project 创建权"（绕过 scheduler）
