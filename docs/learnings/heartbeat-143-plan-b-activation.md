# Learning：派发系统死循环 vs 方案 B 人工恢复

## 🔍 关键发现

### 派发系统的根本问题：自我修复死循环

**设计缺陷**：派发系统的自我修复机制进入了一个无法自拔的死循环。

```
Day 0 (2026-03-19 04:15 UTC):
  派发系统宕机 → 创建修复 Task
  ↓
Day 0+ 几分钟:
  修复 Task 需要通过派发系统派发
  但派发系统已宕机 → 无法派发修复 Task
  ↓
Day 1+ (20+ 小时无恢复):
  修复 Task 永久被卡在 queued 状态
  派发系统无法执行自我修复

结果: 自我修复完全失效，需要人工干预
```

**为什么这是架构问题而非临时故障**：
- 修复路径本身依赖故障的系统 → 循环依赖
- 故障发生后无其他恢复通道 → 单点故障
- 20+ 小时无自我恢复 → 说明系统超出了设计容错范围

### 方案 B 的核心洞察：绕过派发系统的人工路径

**关键决策**：不等待派发系统自动恢复，而是直接创建 Initiatives + Tasks 的代码化表示。

```
派发系统故障 (20+ 小时)
  ↓ 不能依赖派发系统恢复
人工创建 9 个 Initiative 文档 + 5 个 Task 文档
  ↓ 文档即容器，不依赖派发系统
PR 合并到主分支
  ↓ 代码库成为单一真实源
Brain API 批量创建 Initiatives + Tasks
  ↓ 派发系统恢复时自动消费
或主管直接派发（/dev）
  ↓ 无需派发系统介入
```

**为什么有效**：
- 脱离对派发系统的依赖 → 降低耦合度
- 文档形式 → 可版本控制 + 可审查 + 可回滚
- 人工决策 + 自动派发 → 最小化人工但保证执行

## 💡 系统架构改进点（未来建议）

### 1. 自我修复需要独立通道

**问题**：修复任务依赖故障的派发系统
**解决方案**：
```
选项 A：修复任务直接在引导加载器中运行（无需派发）
选项 B：维护一个"紧急派发"通道（基于不同的队列）
选项 C：支持周期性的 health-check + 自动重启（但需幂等）
```

### 2. 分解数据的持久化与同步

**问题**：Initiatives 数据只在 Brain 内存中，故障时丢失
**解决方案**：
```
- 分解数据存储在 PostgreSQL，Brain 定期同步
- Initiatives 变化记录到事件日志
- 支持从事件日志恢复状态
```

### 3. 派发系统的可观测性提升

**问题**：派发故障后无法诊断（缺少日志、metrics）
**解决方案**：
```
- 每个派发事件记录到审计日志
- 提供实时派发队列查询 API（不依赖派发系统本身）
- 故障时支持"干模式"（dry-run 显示会派发什么，但不真正派发）
```

## 🎯 对 zenithjoy 部门的启示

### 文件驱动的工作流更健壮

**经验**：通过 PR + 代码库来驱动 OKR 分解，比依赖派发系统更靠谱

```
传统模式（派发系统驱动）:
  OKR → 派发系统 → 生成 Initiatives → 派发 Task
  问题: 派发系统故障 → 全流程卡住

新模式（文件驱动）:
  OKR → 手工创建 Initiatives 文档 → PR 合并 → Brain API 创建 → 派发 Task
  优势: 派发系统故障 ≠ 分解停滞（文档先行）
```

### 部门主管的权限很关键

**启示**：给部门主管"直接派发"的权限（/dev），而不是永远依赖 Cecelia 派发系统

```
当前（需要派发系统）:
  主管创建 Task → 派发系统 → 派发给 Executor
  问题: 派发系统故障 → 主管无法派发

改进（支持直接派发）:
  主管创建 Task → 主管 /dev 派发 → Executor
  优势: 派发系统故障时仍可派发
```

## ✅ 本轮应用的具体做法（可复制）

### Step 1：快速诊断故障
- 3 个关键查询（派发 endpoint、修复任务状态、分解数据）
- 15 分钟内确认死循环 → 激活应急方案

### Step 2：人工创建 Initiatives 框架
- 按 KR 分类创建 YAML 文档（9 个）
- 包含完整的 acceptance_criteria
- 30 分钟完成 → 可立即进入 PR 流程

### Step 3：创建可执行的 Task 清单
- 从 Initiatives 自动推导 Task（5 个）
- 带优先级 + deadline → 准备派发
- 20 分钟完成

### Step 4：写实时日报供上级决策
- OKR 进度对比（系统数据 vs 实际）
- 瓶颈分析 + 时间压力评估
- 激活决策：继续等 vs 启动方案 B
- 45 分钟完成 + 充分论证

**总耗时**：约 2 小时，从故障诊断到方案 B 完全启动

## 📌 核心教训（通用）

> **当自动化系统故障时，人工流程比等待自动修复更快。**

1. **自我修复有设计边界**：修复路径本身不能依赖故障的系统
2. **文件驱动比系统驱动更健壮**：代码库是最可靠的单一真实源
3. **关键决策需要人脑**：主管的判断（派发系统何时恢复）比算法预测更准确
4. **应急权限要给对人**：部门主管的直接派发权优于完全依赖派发系统
